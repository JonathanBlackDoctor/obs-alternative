# SilentStream E2E 테스트 시나리오 (Phase 6)

> 모든 시나리오는 **로컬 Windows 10/11 x64** 에서 수행한다.
> 사전 준비: `%AppData%\SilentStream\client_secret.json`(데스크톱 앱 OAuth),
> 라이브 스트리밍이 활성화된 YouTube 테스트 채널, FFmpeg 번들.

## S1. 클린 설치
1. `SilentStream-Setup-*.exe` 실행 → EULA 동의 → 자동 시작 방식 선택(시작 프로그램) → 녹화 폴더 선택.
2. **확인**: 설치 완료 후 `%AppData%\SilentStream\config.json` 에 선택 사항 반영, 최초 실행 시 동의 화면 1회 표시.

## S2. 최초 인증 + 수동 송출
1. 첫 실행 → 동의 → 브라우저 OAuth 로그인.
2. `Ctrl+Shift+F12` → 제어판 → "시작".
3. **확인**: 상태 배지 연결 중(🟡) → LIVE(🟢), 좌상단 9px 박스 색 동기화, YouTube Studio에 Unlisted 라이브 생성, 녹화 폴더에 `SilentStream_REC_*.mp4` 증가, 성능 패널에 bps/fps 표시.

## S3. 재부팅 자동 시작
1. PC 재부팅 → 로그인 → 아무 조작도 하지 않음.
2. **확인**: 약 30초 후 자동 송출+녹화 시작(🟡→🟢), 작업표시줄/Alt+Tab에 창 없음, 9px 박스만 표시. 새 브로드캐스트가 생성됨(이전 것 재사용 아님).

## S4. 네트워크 단절·복구 (녹화 무중단)
1. LIVE 상태에서 랜선 분리(또는 방화벽으로 RTMP 차단).
2. **확인**: 박스 🔴(재시도), 로그에 지수 백오프(1→2→…→60초 캡) 재시도 기록, **녹화 파일은 계속 증가**.
3. 네트워크 복구.
4. **확인**: 자동 재연결 → 🟢 복귀(필요 시 새 브로드캐스트 생성).

## S5. 인코더 사망 워치독
1. LIVE 상태에서 작업 관리자로 ffmpeg.exe 강제 종료.
2. **확인**: 워치독이 감지 → 파이프라인 재구성 → 같은 세션의 `_part2` 파일로 녹화 재개 → 🟢 복귀.

## S6. 정상 종료
1. LIVE 상태에서 PC 종료(또는 로그오프).
2. **확인**: mp4가 손상 없이 재생됨, YouTube 브로드캐스트가 "종료됨(complete)" 상태, 다음 부팅에 S3 재현.

## S7. 비정상 종료 내성
1. LIVE 상태에서 전원 강제 차단(또는 프로세스 트리 강제 종료).
2. **확인**: 재부팅 후 직전 mp4가 `frag_keyframe+empty_moov` 덕에 마지막 프래그먼트까지 재생 가능.

## S8. 보존·용량 정책 (시간 가속 모킹)
1. 녹화 폴더에 생성 시각을 8일 전으로 조작한 더미 `SilentStream_REC_*.mp4` 배치.
2. 용량 한도를 작게(예: 1GB) 설정 후 앱 시작.
3. **확인**: 시작 시 1회 정리로 만료 파일 삭제, 한도 초과 시 오래된 파일부터 삭제, 현재 세션 파일은 보호됨. (단위 테스트로도 검증됨)

## S9. 오디오 장치 시나리오
1. LIVE 중 마이크 분리 → **확인**: 시스템 오디오만으로 송출 지속, 로그 경고.
2. 마이크 재연결 + 제어판에서 장치 선택 → **확인**: 믹스 복귀.
3. 슬라이더 조절 → **확인**: 송출/녹화 음량 실시간 반영.

## S10. 단축키·UI
1. `Ctrl+Shift+F12` 토글, 설정에서 `Ctrl+Alt+F9` 로 변경 → 저장.
2. **확인**: 새 단축키로 토글, 이전 단축키 해제, config.json 반영, 로그 뷰어 1000줄 제한.

## S11. 업데이트
1. 구버전 설치 → 릴리스 피드에 신버전 게시.
2. **확인**: 6시간 주기(또는 재시작 시) 감지·적용, 다음 시작 시 신버전 실행.

## S12. 제거
1. 제어판 → 프로그램 제거.
2. **확인**: 시작 프로그램 Run 값/작업 스케줄러 작업 삭제, 로그 폴더 정리, 녹화 파일과 설정은 보존.
